# Demo Mode Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the www sign-in / request-access buttons with a single "Try the demo" button that mints a 2h temporary workspace via better-auth and self-cleans every 30 minutes.

**Architecture:** New `+feature:demo` bounded context at `apps/api/src/demo/` that (a) handles `POST /v1/demo/start` to create a pre-seeded workspace and issue a better-auth session cookie, (b) runs a cron job to delete expired demos. Companion UI changes in `apps/www` (button swap) and `apps/web-tanstack` (banner + expired-demo route). Every file and every cross-cutting edit is fenced with `// +feature:demo` / `// -feature:demo` so the CLI's strip pass removes the feature entirely.

**Tech Stack:** Hono, better-auth 1.6.x, Prisma 7, TanStack Start (web-tanstack), Vite (www), vitest, graphile-worker (dev) / qstash (prod candidate).

**Spec reference:** `docs/2026-04-24-demo-mode-design.md`.

---

## Ground rules for the implementer

- **Fences.** Every new file in `apps/api/src/demo/` opens with `// +feature:demo` on the first line after imports are written (actually: wrap the whole file body between `// +feature:demo` at top and `// -feature:demo` at bottom). Every edit to a shared file (composition.ts, router.ts, schema.prisma, better-auth.ts, etc.) must insert fenced blocks — never unfenced demo code.
- **Directives.** The `// +feature:X ... // -feature:X` pattern is a line-range delete by the CLI generator. Keep opening and closing fences on their own lines. Never nest a `+feature:demo` fence inside a broken fence — the closing must match.
- **Commit after every task.** Each task ends with a `git add` + `git commit`. Never combine tasks into one commit.
- **No emojis.** Global CLAUDE.md forbids them in code and commits.
- **ast-grep.** Global CLAUDE.md says to use `sg` for syntax-aware searches. Prefer over `grep` when searching for function calls / type references.
- **Commit style.** Lowercase imperative subject (match `git log` — e.g. `feat: ...`, `docs: ...`).

import { Hono } from "hono";
import { requireSession } from "../middleware/session.ts";
import type { HonoEnv } from "../middleware/container.ts";
import { userToDTO } from "@/interfaces/mappers.ts";

export const me = new Hono<HonoEnv>();

me.get("/", async (c) => {
  const container = c.get("container");
  const session = requireSession(c);
  const { user, workspaces } = await container.services.getMe.resolveByUserId(session.userId as never);

      id: w.workspaceId,
      memberId: w.id,
      slug: w.slug,
      name: w.name,
      roleName: w.roleName,
      roleSystemKey: w.roleSystemKey,
    })),

The repo has no such harness yet. The smoke test in Task 14 covers the same end-to-end behavior manually. Fence density is already high in these files. Not acting on it here — the spec is explicit about fencing, and unbundling would be a separate refactor.
